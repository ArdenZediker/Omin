# 知识库 RAG 流程说明

这份文档说明的是本项目里“知识库问答”是怎么跑起来的。写法尽量保持技术文档风格，但会尽量用容易理解的表达。

## 1. 一句话理解

RAG 的核心流程可以理解成：

1. 把文件导入知识库
2. 把文件拆成适合检索的小段
3. 给每个小段生成向量，方便做语义检索
4. 用户提问时，先从知识库里找相关片段
5. 把相关片段和用户问题一起交给模型回答

简单说就是：

- 导入阶段负责“整理资料”
- 检索阶段负责“找到资料”
- 回答阶段负责“用资料回答”

## 2. 导入阶段

知识库入口在知识库页面。用户上传文件后，前端会先做基础解析，再把内容交给后端保存。

### 2.1 前端会做什么

前端会尽量先提取可读文本：

- `markdown` / `text` 文件直接读取文本
- `docx` 先转成文本
- `pdf` 先提取文本
- 其他类型文件可能没有可直接提取的正文，但原文件仍然会被保存

除了正文，前端还会一起传给后端这些信息：

- 文件名
- 文件来源路径
- MIME 类型
- 扩展名
- 预览类型
- 缩略图
- 标签
- 收藏状态

### 2.2 后端会做什么

后端收到内容后，会做这些处理：

1. 检查知识库是否存在
2. 标准化文件信息
3. 按规则把文档切分成多个 chunk
4. 为每个 chunk 生成向量
5. 把文档元数据和 chunk 一起写入数据库
6. 如果传了原文件字节，还会把原文件保存到本地磁盘

导入完成后，知识库页面就能看到这份文档，并可用于后续检索。

## 3. 文档切分

整篇文档通常不适合直接拿去检索，所以系统会先把它拆成更小的片段。

### 3.1 切分目标

切分不是简单按固定字数砍开，而是尽量让每一段保持完整语义。这样做的原因是：

- 检索更准
- 命中的内容更短
- 更适合作为模型上下文

### 3.2 系统怎么判断文档结构

切分前，系统会先看文档有没有明显结构。判断依据主要是这些：

- 有没有标题和小标题
- 有没有章节编号
- 有没有明显的段落分隔
- 有没有列表、项目符号、表格
- 是否存在比较明确的章节边界

如果这些特征比较明显，系统会认为这篇文档“结构清楚”。
如果看起来像一大段连续文本，没有明显层次，就会认为“结构不清楚”。

### 3.3 切分方式

系统会按优先级选择切分方式：

1. 有标题结构的，优先按标题切
2. 结构比较明显但不是标题文档的，按段落和自然边界切
3. 结构不明显的，按标点、句子和长度切
4. 如果前面的结果不合格，会退回到更保守的方式

### 3.4 切分后会检查什么

系统不是无条件接受切分结果，还会检查：

- chunk 不能为空
- chunk 不能太长
- 不能拆出大量特别小的碎片
- 整体内容必须是有效的

如果某种切分方式结果不好，系统会自动换一种方式再试。

## 4. 向量化

切分完成后，系统会给每个 chunk 生成一个向量。

### 4.1 向量化是什么

可以简单理解成“把文字变成一组数字表示”。
这组数字不是给人看的，而是给系统用来比较相似度的。

每个 chunk 都会有自己的“数字特征”，用户的问题也会变成类似的数字特征。
系统通过比较这些特征，判断哪段内容和问题最接近。

### 4.2 为什么需要向量化

如果只靠关键词，容易出现这些问题：

- 字面不一样，但意思很接近
- 用户问法比较口语，资料写法比较正式
- 同一个问题在不同文档里表达方式不同

向量化的作用，就是补上“语义相似”的判断能力。

### 4.3 什么时候生成向量

导入文档时，系统会尝试给每个 chunk 生成向量。

如果 embedding 配置不可用，系统不会直接失败，而是让这部分内容继续以文本方式保存。
这意味着：

- 能生成向量时，检索会更准
- 不能生成向量时，系统仍然可以工作，只是更依赖关键词匹配

## 5. 检索阶段

用户开始提问时，系统会先查知识库，而不是直接把问题发给模型。

### 5.1 查询来源

查询一般来自两种情况：

- 用户显式指定的知识库问题
- 最近一条用户消息

如果没有可用查询内容，系统不会进入知识库检索。

### 5.2 检索时看什么

检索会综合考虑：

- 问题里的关键词
- chunk 文本里的词语命中情况
- chunk 标题
- 原始文档名
- 如果有向量，还会考虑语义相似度

可以理解成它不是只看“字面是否一样”，还会看“意思像不像”。

### 5.3 检索结果怎么排序

系统会把候选 chunk 按相关度排序，相关度高的排前面。
如果分数差不多，还会参考一些附加信息，比如：

- 访问次数
- 创建时间

最后只取前几条结果作为本次知识库上下文。

## 6. 检索结果如何进入模型

检索出来的内容不会直接混进用户消息里，而是作为一段独立的参考上下文传给模型。

这样做的原因是：

- 模型更容易区分“用户说的话”和“参考资料”
- 可以明确告诉模型，这些内容只能作为参考
- 方便后续展示引用来源

系统会把命中的 chunk 组织成一个知识块，里面通常包含：

- 来源文档名
- 所属知识库
- chunk 编号
- 片段标题
- 片段正文
- 命中分数
- 标签和收藏等元信息

然后再和原始对话一起发给模型。

## 7. embedding 配置

知识库的向量化配置由设置页管理。

### 7.1 配置内容

主要包括：

- 是否启用 embedding
- 使用哪个 provider
- 使用哪个 embedding 模型

### 7.2 配置影响范围

这套配置只影响知识库文档和查询的向量化，不影响普通聊天模型。
也就是说：

- 聊天模型可以正常独立使用
- 知识库向量化是另一套单独配置

### 7.3 配置不可用时怎么办

如果 provider 没配好、API key 缺失，或者请求失败，系统会自动降级：

- 文档仍然可以导入
- chunk 仍然会保存
- 检索仍然可以运行
- 只是更偏向关键词匹配

## 8. 失败和降级

这套流程的设计比较保守，目标是“不要因为知识库问题阻断聊天”。

常见降级路径有：

- 向量生成失败，不影响导入
- 查询向量生成失败，不影响检索
- 检索接口失败，直接退回普通聊天
- 没有查到相关内容，直接走无知识库回答

## 9. 文档详情页怎么看

知识库里的文档详情页会分成两块看：

- 原始格式预览
- 结构化后的 chunk 列表

PDF 的展示会尽量保留“原文件效果”，同时在下面给出提取出来的原文，方便核对和搜索。

这也是为了尽量接近 WeKnora 的思路：

- 先看文件原貌
- 再看系统提取的文本
- 最后看分片结果

## 10. 关于是否必须开启向量

这个项目更适合“两种都支持，但默认允许不开启”。

原因很直接：

- 有向量时，效果更好
- 没有向量时，功能还能用，不至于卡死在配置上
- 用户可以先导入资料，再补配 embedding

但要注意一点：

- 如果一批文档已经按“无向量模式”导入，后来又开启向量，最好不要把旧数据和新数据混成一团
- 比较稳妥的做法是给每个文档加一个“向量化状态”，必要时允许手动重建

这样用户就能清楚知道：

- 哪些文档已经向量化
- 哪些文档还没向量化
- 什么时候需要重新生成

## 11. 你现在最该理解的结论

RAG 不是“把整个文档扔给模型”。
它更像是：

“先把资料整理好，再根据问题从资料里找最相关的片段，最后把这些片段交给模型回答。”

这也是本项目里知识库的核心工作方式。
